# 1時間アイディアソン ワークショップ 最終設計（v3）

## 概要

AWS ML Enablement Workshopの良いところを参照しつつ、Amazon Q Developerとの壁打ちで個人ワーク中心に90分でアイディアを膨らませ、モックを作成し、チームビルディングまで行うワークショップ。

**コンセプト:**
- Working Backwardsではなく、独自のアイディアソン
- Working Backwardsの良いところは継承
- 参加者のインスピレーションの種をAIで膨らませる体験

## イベント情報

### 参加者
- **合計20人**（オンライン10人 + オフライン10人）
- ほぼエンジニア、一部ビジネス職
- Amazon Q Developer初心者
- **前イベントでPython 3.11 + Strands Agents体験済み**

### 前提環境
- Python 3.11 + uv
- Amazon Q Developer
- VS Code
- Strands Agents実装経験あり
- 前イベントのリポジトリ: https://github.com/ist-whitebox/ai-hackathon-2025-hands-on

### 本イベントのリポジトリ
- このリポジトリをcloneして使用
- プロンプトテンプレート、ガイドを含む

## タイムテーブル

詳細なタイムテーブルは **[your_work/WORKSHOP.md](../your_work/WORKSHOP.md)** を参照してください。

## 各セクションの詳細

### 0-5分: イントロ

**目的:** ワークショップのゴールと進め方を共有

**内容:**
- 今日のゴール
  1. AIとのアイディア膨らませ方を体験
  2. モックを作る（Strands Agents活用）
  3. チームを作る（マッチング）
- 3ステップの説明
  - Step1: 顧客理解
  - Step2: アイディア発明
  - Step3: ビジネス化
- 前回のStrands Agents環境を使うことを確認
- このリポジトリをcloneしてもらう

### 5-10分: 環境確認

**目的:** 前回環境が使えることを確認

**チェック項目:**
- [ ] Amazon Q Developerが起動する
- [ ] `q chat`でチャットができる
- [ ] 前回のPython環境が残っている
- [ ] このリポジトリをcloneした

**トラブル時:**
- 運営・チューターに手を挙げる
- 前回のハンズオン資料を参照

### 10-40分: アイディア膨らまし（30分）

**目的:** 身近な課題からビジネスアイディアを作成

#### Step1: 顧客理解 🤖

**ワーク:**
1. 身近な課題をQ Developerに相談
2. プロンプトテンプレートを使って分析

**プロンプト:** `prompts/step1_customer_understanding.md` を参照

**アウトプット:**
- ペルソナ（顧客像）
- 行動の時系列マップ
- 改善すべき課題Top3

#### Step2: アイディア発明 🤖

**ワーク:**
1. 参考事例を収集
2. 事例を組み合わせてアイディア発明
3. 最有望案を選択

**プロンプト:** `prompts/step2_idea_invention.md` を参照

**アウトプット:**
- 参考事例5つ
- 発明したアイディア3つ
- 選択した最有望案1つ

#### Step3: ビジネス化 🤖

**ワーク:**
1. ビジネスモデルキャンバス作成（オプション）
2. ビジネスモデルの検証（オプション）
3. 簡易PR/FAQ作成（必須）

**プロンプト:** `prompts/step3_business_model.md` を参照

**重要:**
- ビジネスモデルキャンバスは時間があればやる（オプション）
- PR/FAQは必須（モック作成の元になる）

**アウトプット:**
- ビジネスモデルキャンバス（オプション）
- 簡易PR/FAQ（必須）

### 40-65分: モック作成

**目的:** Strands Agentsを活用してポン出しモックを作成

#### モックの優先順位

1. 🥇 **Python + Flask/FastAPI + 画面 + Strands Agents処理**（推奨）
2. 🥈 **Node.js/Next.js + 画面 + 処理**
3. 🥉 **HTML + JavaScript + Python**
4. 🏅 **HTML + JavaScript（入力→出力）**

**プロンプト:** `prompts/mock_creation.md` を参照

**重要:**
- モックは「ポン出し」レベルでOK
- 完璧を求めない
- 動けばOK
- 時間がない場合はPR/FAQだけでもOK

### 60-65分: 提出

**目的:** 仕様書・モック・想いを提出

#### 提出フォーム（Google Form / Slack）

1. **名前**
2. **仕様書（PR/FAQ）**
   - テキスト貼り付け or ファイルアップロード
3. **ビジネスモデルキャンバス**（オプション）
   - 作成した人のみ
4. **モックのスクリーンショット or 動画**
   - 画面キャプチャ
   - 動作デモ動画（30秒程度）
   - なくてもOK
5. **実現したいこと（想い）**
   - 抽象的でOK
   - 「こんな世界を作りたい」
   - 100文字程度
6. **チームメンバー募集**
   - Yes / No
   - Yesの場合、どんな人を募集しているか

**提出タイミング:**
- モック作成中（60分頃）から記入開始
- 65分までに提出

### 65-70分: 運営準備

**運営がやること:**
1. 提出された一覧を作成
2. 発表者を選定（3-5個）
   - 運営・チューターが気になったもの
   - 参加者に「気になるものある？」と聞く
   - 「我こそは！」という人を募る
3. 発表順を決定
4. 全員の「想い」一覧を作成

### 70-85分: 共有 + チームビルディング（15分）

**目的:** アイディアを共有し、チームマッチング

#### 発表（3-5個 × 3分 = 9-15分）

**発表者がやること:**
1. 画面共有
2. 仕様書（PR/FAQ）を見せながら説明
3. モックをデモ（あれば）
4. 募集の呼びかけ（募集する場合）

**発表フォーマット:**
- 課題と顧客（30秒）
- 解決策の概要（1分）
- モックデモ（1分）
- 募集の呼びかけ（30秒）

#### チームビルディング（残り時間）

**運営がやること:**
1. 全員の「想い」一覧を画面共有
2. 自由交流タイム
3. その場でマッチング

**参加者がやること:**
- 気になるアイディアに声をかける
- 一緒にやりたい人を探す
- Slackで後日連絡することを約束

### 85-90分: Next Step

**内容:**
- Slackでの継続マッチング案内
  - 全アイディア一覧を後日公開
  - 募集・応募はSlackで継続
- 今後の展開
  - 次回ハッカソンでの実装
  - チームでの開発継続

## 成功のポイント

### 時間管理
- タイマーを使って厳密に管理
- 各ステップ2分前にアラート
- 遅れている人がいても次に進む

### サポート体制
- 運営 + チューター複数名
- 「困ったら手を挙げる」ルール明示
- オンライン参加者もチャットでサポート

### ゴール設定
- **最低限のゴール:** PR/FAQ作成まで
- **理想のゴール:** モック構築まで
- **ボーナス:** ビジネスモデルキャンバス + Strands Agents活用

### 雰囲気作り
- 「ポン出しでOK」を強調
- 「完璧を求めない」
- 「途中まででもOK」
- 「AIとの壁打ちを楽しむ」
- 「チームビルディングが本番」

## よくある課題と対策

| 課題 | 対策 |
|------|------|
| Q Developerが動かない | 前回環境を使う、チューターサポート |
| プロンプトがうまくいかない | テンプレートを提供、代替案を用意 |
| モック構築が間に合わない | PR/FAQ作成までで区切る、簡易版に切り替え |
| Strands Agentsでエラー | 前回のサンプルコードを参考に |
| 時間オーバー | 各ステップ2分前にアラート、次に進む |
| ビジネスモデルキャンバスが難しい | オプションなのでスキップしてOK |

## トラブルシューティング

### 環境が動かない
1. 前回のハンズオン資料を参照
2. チューターに相談
3. 最悪、HTML+JSで進める

### プロンプトの出力が期待と違う
1. プロンプトを修正して再実行
2. 追加質問で深掘り
3. 手動で修正してもOK

### モックが作れない
1. 簡易版（HTML+JS）に切り替え
2. PR/FAQだけでもOK
3. 発表時に「構想」として説明

## まとめ

この90分版アイディアソンは、以下を実現します：

**体験できること:**
- AIとのアイディア膨らませ方の体験
- Amazon Q Developerとの効果的な壁打ち
- Strands Agentsを活用したモック作成
- チームビルディング・マッチング

**持ち帰れるもの:**
- PR/FAQ（仕様書）
- ビジネスモデルキャンバス（オプション）
- ポン出しモック
- チームメンバー候補
- 次のステップへの道筋

**重要な点:**
- 個人ワークで各自のペースで深掘り
- テンプレートプロンプトで迷わない
- ビジネスモデルキャンバスはオプション
- モックは「ポン出し」でOK
- チームビルディングが本番
- Slackで継続マッチング

**Working Backwardsとの関係:**
- Working Backwardsではなく、独自のアイディアソン
- Working Backwardsの良いところ（顧客起点、問いを立てる）は継承
- ブランドを傷つけない独立したワークショップ
